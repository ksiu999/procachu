# Детализация бизнес-процессов: альтернативные сценарии и статусы

> 💡 **Дополнение к Документу 2:** Альтернативные сценарии, статусы сущностей, SLA и требования к безопасности

---

## 1. Статусы сущностей и правила переходов

### 1.1 Статусы велосипеда

| Статус          | Описание                            | Разрешённые переходы                                      |
| --------------- | ----------------------------------- | --------------------------------------------------------- |
| **Available**   | Доступен для аренды                 | → Booked, → Maintenance                                   |
| **Booked**      | Зарезервирован клиентом на 15 минут | → Available (истечение брони), → Active (старт аренды)    |
| **Active**      | В аренде у клиента                  | → Available (возврат на станцию), → Maintenance (поломка) |
| **Maintenance** | На ремонте                          | → Available (ремонт завершён)                             |

### 1.2 Статусы аренды

| Статус             | Описание                            | Триггер перехода               |
| ------------------ | ----------------------------------- | ------------------------------ |
| **Pending**        | Бронь создана, ожидает старта       | Клиент нажал "Забронировать"   |
| **Active**         | Аренда начата, идёт тарификация     | Клиент разблокировал замок     |
| **Completed**      | Успешно завершена, средства списаны | Возврат на станцию подтверждён |
| **ForceCompleted** | Принудительно завершена (23:00)     | Системный таймер               |
| **Cancelled**      | Отменена (бронь истекла)            | Истечение 15 минут без старта  |
| **Incident**       | Инцидент (поломка, спор)            | Клиент сообщил о проблеме      |

### 1.3 Статусы платежа

| Статус            | Описание                                      |
| ----------------- | --------------------------------------------- |
| **Pending**       | Платёж инициирован, ожидает подтверждения     |
| **Success**       | Успешно проведён                              |
| **Failed**        | Отклонён (недостаточно средств, ошибка шлюза) |
| **Refunded**      | Возвращён клиенту                             |
| **PartialRefund** | Частичный возврат                             |

### 1.4 Статусы тикета поддержки

| Статус         | Описание             | SLA                       |
| -------------- | -------------------- | ------------------------- |
| **New**        | Новый тикет          | —                         |
| **InProgress** | В работе у оператора | Время реакции: ≤ 15 минут |
| **Resolved**   | Решён                | Время решения: ≤ 24 часа  |
| **Closed**     | Закрыт               | —                         |

## 2. Альтернативные и ошибочные сценарии

### BP-01: Регистрация и аутентификация

**Альтернативный сценарий А1: Пользователь уже зарегистрирован**

- Система распознаёт номер телефона
- Отправляет SMS-код
- Пользователь входит без повторной регистрации

**Ошибочный сценарий E1: SMS не доставлен**

- Пользователь не получает код в течение 60 секунд
- Система предлагает: "Запросить код повторно" или "Использовать голосовой вызов"
- После 3 неудачных попыток — блокировка на 15 минут (защита от фрода)

**Ошибочный сценарий E2: Неверный код**

- Пользователь вводит неправильный код
- Система показывает: "Неверный код, осталось 2 попытки"
- После 3 ошибок — временная блокировка

---

### BP-03: Бронирование велосипеда

**Альтернативный сценарий А1: Велосипед уже забронирован**

- Клиент выбирает станцию, но все велосипеды забронированы
- Система показывает: "Нет свободных велосипедов. Ближайшая свободная станция — [название], [расстояние]"
- Предлагает подписаться на уведомления об освобождении

**Ошибочный сценарий E1: Бронь истекла**

- Клиент не начал аренду в течение 15 минут
- Система автоматически снимает бронь
- Push-уведомление: "Ваша бронь истекла. Велосипед снова доступен для других"
- Штрафов нет (бронь бесплатная)

**Ошибочный сценарий E2: Клиент отменил бронь**

- Клиент нажал "Отменить бронь" до начала аренды
- Велосипед возвращается в статус Available
- Бронь считается отменённой, списаний нет

---

### BP-04: Старт аренды (разблокировка)

**Альтернативный сценарий А1: Bluetooth недоступен**

- Приложение не может подключиться к замку по BLE
- Система предлагает альтернативу: "Отсканировать QR-код на замке"
- Клиент открывает камеру, сканирует QR
- Система проверяет: QR соответствует забронированному велосипеду?
- Если да — отправляет команду разблокировки через сервер

**Ошибочный сценарий E1: Замок не отвечает**

- Команда разблокировки отправлена, но замок не открылся в течение 10 секунд
- Система показывает: "Не удалось разблокировать замок. Обратитесь в поддержку"
- Автоматически создаётся тикет в CRM с приоритетом "Высокий"
- Клиенту предлагается выбрать другой велосипед (если есть на станции)

**Ошибочный сценарий E2: Велосипед уже в аренде**

- Клиент пытается разблокировать велосипед, который уже арендован другим пользователем
- Система показывает: "Этот велосипед уже используется. Выберите другой"
- Карта обновляется, показывая актуальные статусы

---

### BP-05: Завершение аренды

**Альтернативный сценарий А1: GPS-погрешность (клиент рядом, но система не видит возврат)**

- Клиент вернул велосипед на станцию, но GPS показывает расстояние > 20 метров
- Система не может автоматически завершить аренду
- Клиент нажимает "Завершить аренду вручную"
- Система запрашивает фото велосипеда на станции (опционально)
- Оператор проверяет фото и завершает аренду вручную (SLA: ≤ 15 минут)

**Ошибочный сценарий E1: Платёж не прошёл**

- Система попыталась списать средства, но платёжный шлюз вернул ошибку "Недостаточно средств"
- Аренда остаётся в статусе Active
- Push-уведомление: "Не удалось списать средства. Пополните баланс в течение 24 часов"
- Если средства не поступили за 24 часа — тикет в поддержку, возможная блокировка аккаунта

**Ошибочный сценарий E2: Клиент не вернул велосипед до 23:00**

- Время 23:00, велосипед всё ещё в аренде
- Система автоматически завершает аренду (статус ForceCompleted)
- Списывает средства по тарифу + штраф 500 рублей за нарушение правил
- Push-уведомление: "Аренда принудительно завершена. Списано: [сумма] + штраф 500 руб."
- Если велосипед не возвращён на станцию — создаётся тикет для курьера (SLA: вызов в течение 1 часа)

**Ошибочный сценарий E3: Клиент выехал за МКАД**

- GPS показывает, что клиент выехал за пределы МКАД
- Система отправляет push-уведомление: "Вы выехали за зону покрытия. Вернитесь в пределах МКАД, иначе аренда будет принудительно завершена через 30 минут"
- Если клиент не вернулся за 30 минут — автозавершение + штраф 1000 рублей

---

### BP-06: Обработка инцидентов (поломка)

**Альтернативный сценарий А1: Клиент сообщил о поломке, но хочет продолжить поездку**

- Клиент нажал "Сообщить о поломке", но выбирает "Продолжить поездку"
- Система предупреждает: "Если поломка критичная (тормоза, руль), рекомендуем прекратить поездку"
- Если клиент продолжает — аренда идёт в обычном режиме
- После завершения — тикет автоматически создаётся, велосипед помечается как "Требует проверки"

**Ошибочный сценарий E1: Не удалось создать тикет**

- Клиент отправил отчёт о поломке, но сервер недоступен
- Приложение сохраняет отчёт локально
- При восстановлении связи — автоматическая отправка
- Push-уведомление: "Ваш отчёт будет отправлен при восстановлении связи"

---

### BP-09: Обработка возвратов средств (оператор)

**Альтернативный сценарий А1: Частичный возврат**

- Оператор решает вернуть не полную сумму, а часть (например, 50% из-за технической проблемы)
- Оператор вводит сумму возврата вручную
- Система проверяет: сумма ≤ стоимости аренды?
- Если да — проводит частичный возврат
- В истории остаётся запись: "Частичный возврат: [сумма]"

**Ошибочный сценарий E1: Возврат не прошёл**

- Оператор инициировал возврат, но платёжный шлюз вернул ошибку
- Тикет остаётся в статусе InProgress
- Оператор получает уведомление: "Возврат не удался. Попробуйте позже или обратитесь в финансовый отдел"
- Система логирует ошибку для дальнейшего разбора

**Ограничение безопасности:** Оператор может вернуть средства только в пределах стоимости аренды. Возврат суммы, превышающей стоимость поездки, требует подтверждения от руководителя (двухфакторная авторизация).

## 3. SLA (Service Level Agreement)

| Процесс                                       | Метрика                                                   | Значение        | Ответственный      |
| --------------------------------------------- | --------------------------------------------------------- | --------------- | ------------------ |
| **Реакция на тикет "Поломка"**                | Время от создания до начала работы оператора              | ≤ 15 минут      | Оператор поддержки |
| **Решение тикета "Поломка"**                  | Время от начала работы до закрытия                        | ≤ 24 часа       | Оператор поддержки |
| **Вывод велосипеда из парка**                 | Время от подтверждения поломки до статуса Maintenance     | ≤ 1 час         | Курьер / Механик   |
| **Возврат велосипеда после ремонта**          | Время от завершения ремонта до статуса Available          | ≤ 2 часа        | Механик            |
| **Обработка возврата средств**                | Время от запроса оператора до зачисления на карту клиента | ≤ 3 рабочих дня | Финансовый отдел   |
| **Доставка курьера (велосипед не возвращён)** | Время от создания тикета до прибытия курьера              | ≤ 1 час         | Курьерская служба  |

## 4. Требования к безопасности и авторизации

### 4.1 Роли и права доступа

| Роль                    | Доступ к функциям                                                                | Ограничения                                                |
| ----------------------- | -------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **Клиент**              | Бронирование, аренда, оплата, история, отчёты о поломках                         | Не может видеть данные других клиентов                     |
| **Оператор поддержки**  | Просмотр тикетов, возвраты (≤ стоимости аренды), ручное завершение аренд         | Не может удалять данные, не может менять тарифы            |
| **Руководитель**        | Все функции оператора + управление тарифами, промокодами, полный возврат средств | Требуется двухфакторная авторизация для критичных операций |
| **CTO / Администратор** | Управление парком, станциями, мониторинг логов, доступ к БД                      | Не может проводить финансовые операции                     |

### 4.2 Безопасность платёжных данных

- ✅ Приложение **не хранит** данные банковских карт (номер, CVV, срок действия)
- ✅ Приложение хранит только **токен платёжного провайдера** (например, `card_token` от ЮKassa)
- ✅ Все транзакции проходят через **сертифицированный платёжный шлюз** (соответствие PCI DSS)
- ✅ Бэкенд не логирует полные данные карт (только последние 4 цифры для идентификации)
- ✅ Передача данных только по HTTPS с TLS 1.2+

### 4.3 Безопасность персональных данных

- ✅ Номер телефона и email хранятся в зашифрованном виде (AES-256)
- ✅ Геолокация передаётся только во время активной аренды
- ✅ После завершения аренды геолокация удаляется (хранится только маршрут для истории)
- ✅ Соответствие ФЗ-152 "О персональных данных" (Россия)

## 5. Интеграции

| Система                                      | Назначение                           | SLA / Ограничения                         |
| -------------------------------------------- | ------------------------------------ | ----------------------------------------- |
| **Платёжный шлюз (ЮKassa / CloudPayments)**  | Обработка платежей, возвраты         | Доступность: 99.9%, время ответа: ≤ 2 сек |
| **SMS-шлюз (SMS.ru / Twilio)**               | Отправка кодов подтверждения         | Доставка: ≤ 60 сек, стоимость: ~2 руб/SMS |
| **Push-уведомления (Firebase / APNs)**       | Уведомления о статусе аренды         | Доставка: ≤ 5 сек                         |
| **Карты (Яндекс.Карты / 2ГИС API)**          | Отображение станций и маршрутов      | Обновление данных: каждые 5 минут         |
| **Bluetooth-замки (проприетарный протокол)** | Разблокировка/блокировка велосипедов | Время отклика: ≤ 3 сек, радиус: ≤ 10 м    |
| **CRM поддержки (Jira / Zendesk)**           | Управление тикетами                  | Синхронизация: real-time                  |
