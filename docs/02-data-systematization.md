# Систематизация данных по проекту «Прокачу»

## 1. Предметная область

**Городской сервис краткосрочной аренды велосипедов «Прокачу»**

**Текущее состояние:**

- Парк: 200 велосипедов
- Инфраструктура: 20 станций в пределах МКАД
- География: Москва (внутри МКАД)
- Режим работы: 06:00 – 23:00
- Текущая платформа: Telegram-бот (MVP)
- Нагрузка: 50-80 аренд/день (обычный день), до 300 аренд/день (пик сезона)
- Пиковые часы: выходные, 12:00 – 18:00

**Планы масштабирования:**

- Увеличение парка до 500 велосипедов
- Расширение до 50 станций
- Выход в один из соседних городов (Подмосковье) в течение 12 месяцев

**Текущие метрики проблем:**

- 132 тикета в поддержку за месяц
- 38% тикетов — критические (недоставка кода разблокировки)
- 20% тикетов — финансовые (ошибочные списания)
- Конверсия в аренду: 60% (падение на 40% из-за 8 шагов в боте)

## 2. Бизнес-процессы

### BP-01: Регистрация и аутентификация пользователя

**Актор:** Клиент
**Триггер:** Первый запуск приложения
**Шаги:**

1. Пользователь вводит номер телефона
2. Система отправляет SMS с кодом подтверждения
3. Пользователь вводит email
4. Система создает учетную запись
5. Пользователь соглашается с офертой и политикой конфиденциальности

**Постусловие:** Пользователь аутентифицирован, может пользоваться приложением

---

### BP-02: Поиск и бронирование велосипеда

**Актор:** Клиент
**Триггер:** Пользователь хочет взять велосипед
**Шаги:**

1. Пользователь открывает карту в приложении
2. Система отображает ближайшие станции с количеством свободных велосипедов
3. Пользователь выбирает станцию
4. Пользователь нажимает «Забронировать»
5. Система резервирует велосипед на 15 минут
6. Система отправляет Push-уведомление с таймером брони

**Постусловие:** Велосипед зарезервирован, доступен только для данного пользователя

**Альтернативный сценарий:** Если пользователь не начал аренду за 15 минут — бронь автоматически снимается

---

### BP-03: Старт аренды (разблокировка велосипеда)

**Актор:** Клиент, Система
**Триггер:** Пользователь подошел к забронированному/выбранному велосипеду
**Шаги:**

1. Пользователь нажимает «Начать аренду» в приложении
2. Приложение устанавливает Bluetooth-соединение с замком велосипеда
3. Система отправляет команду разблокировки
4. Замок открывается (звуковой сигнал)
5. Система фиксирует время начала аренды
6. Система запускает таймер и отображает текущую стоимость поездки

**Постусловие:** Аренда активна, идет начисление стоимости по тарифу

**Исключения:**

- Если Bluetooth не подключился — система предлагает альтернативный QR-код
- Если замок не открылся — создается тикет в поддержку автоматически

---

### BP-04: Активная аренда (мониторинг поездки)

**Актор:** Система, Клиент
**Триггер:** Аренда активна
**Шаги:**

1. Система отслеживает геолокацию пользователя (каждые 10 секунд)
2. Приложение отображает: время поездки, текущую стоимость, карту маршрута
3. За 5 минут до предполагаемого окончания (при суточном тарифе) — Push-уведомление
4. Пользователь может нажать «Сообщить о поломке» в любой момент

**Постусловие:** Пользователь информирован о статусе аренды

---

### BP-05: Завершение аренды (возврат на станцию)

**Актор:** Клиент, Система
**Триггер:** Пользователь вернул велосипед на станцию
**Шаги:**

1. Пользователь паркует велосипед на станции (физически)
2. Пользователь нажимает «Завершить аренду» в приложении ИЛИ система автоматически определяет возврат по геолокации
3. Система проверяет: велосипед находится в зоне станции (радиус 20 метров)
4. Система фиксирует время окончания аренды
5. Система рассчитывает итоговую стоимость по тарифу
6. Система списывает средства с привязанной карты
7. Система отправляет чек на email и Push-уведомление об успешном завершении

**Постусловие:** Аренда завершена, средства списаны, велосипед доступен для других пользователей

**Альтернативный сценарий (автозавершение):**

- Если пользователь забыл завершить аренду, система проверяет геолокацию каждые 5 минут
- При обнаружении, что велосипед на станции более 3 минут — автоматическое завершение
- Push-уведомление: «Аренда завершена автоматически. Списано: XXX руб.»

---

### BP-06: Обработка инцидентов (поломка велосипеда)

**Актор:** Клиент, Оператор поддержки
**Триггер:** Пользователь обнаружил неисправность
**Шаги:**

1. Пользователь нажимает «Сообщить о поломке» в приложении
2. Пользователь выбирает тип поломки (из списка): спустило колесо, сломан руль, не работает тормоз, другое
3. Пользователь добавляет фото (опционально)
4. Система создает тикет в CRM поддержки
5. Система помечает велосипед как «Неисправен» и скрывает из доступных для бронирования
6. Оператор получает уведомление и организует ремонт

**Постусловие:** Велосипед исключен из доступного парка до ремонта

---

### BP-07: Пополнение баланса и оплата

**Актор:** Клиент, Платежный шлюз
**Триггер:** Пользователь хочет пополнить баланс или оплатить поездку
**Шаги:**

1. Пользователь переходит в раздел «Оплата»
2. Пользователь вводит данные карты (или выбирает сохраненную)
3. Система передает данные в платежный шлюз (PCI DSS compliant)
4. Платежный шлюз подтверждает операцию
5. Система зачисляет средства на баланс (при пополнении) ИЛИ списывает за поездку

**Постусловие:** Баланс обновлен, операция зафиксирована в истории

---

### BP-08: Просмотр истории поездок

**Актор:** Клиент
**Триггер:** Пользователь хочет посмотреть историю
**Шаги:**

1. Пользователь открывает раздел «История»
2. Система отображает список поездок (последние сначала)
3. Для каждой поездки: дата, время начала/окончания, длительность, маршрут на карте, стоимость, статус

**Постусловие:** Пользователь получил информацию о прошлых арендах

---

### BP-09: Обработка возвратов средств (оператор)

**Актор:** Оператор поддержки
**Триггер:** Жалоба пользователя на ошибочное списание
**Шаги:**

1. Оператор находит аренду в админ-панели по ID пользователя или номеру телефона
2. Оператор проверяет логи: время начала/окончания, геолокацию
3. Если ошибка подтверждена — оператор нажимает «Вернуть средства»
4. Система создает транзакцию возврата в платежном шлюзе
5. Система отправляет пользователю Push и email о возврате

**Постусловие:** Средства возвращены, тикет закрыт

---

### BP-10: Администрирование системы (CTO)

**Актор:** Технический директор / Администратор
**Триггер:** Регулярный мониторинг или инцидент
**Шаги:**

1. Просмотр дашборда: активные аренды, нагрузка на сервер, ошибки API
2. Управление станциями: добавление/удаление, изменение координат
3. Управление парком: добавление новых велосипедов, исключение неисправных
4. Просмотр логов интеграций (замки, платежный шлюз, SMS-шлюз)
5. Настройка тарифов и промокодов

**Постусловие:** Система актуализирована, инциденты устранены

## 3. Роли

| Роль                                  | Описание                                   | Права доступа                                                                                |
| ------------------------------------- | ------------------------------------------ | -------------------------------------------------------------------------------------------- |
| **Клиент**                            | Пользователь, арендующий велосипед         | Регистрация, поиск станций, бронирование, аренда, оплата, просмотр истории, отчет о поломках |
| **Оператор поддержки**                | Сотрудник службы поддержки                 | Просмотр тикетов, возвраты средств, ручное завершение аренд, связь с клиентами               |
| **CTO / Администратор**               | Технический директор или назначенный админ | Управление парком и станциями, настройка тарифов, мониторинг системы, доступ к логам         |
| **Платежный шлюз** (внешняя система)  | Внешний сервис обработки платежей          | Прием платежных данных, проведение транзакций, возвраты (API)                                |
| **Bluetooth-замок** (внешняя система) | Устройство на велосипеде                   | Прием команд разблокировки, отправка статуса (открыт/закрыт)                                 |
